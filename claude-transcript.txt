  ---
  Feature Plan

  1. User enters their income
  2. User adds expenses (name + amount), which appear in a list
  3. User can remove individual expenses
  4. App shows: total expenses, remaining balance
  5. Balance turns green when positive, red when zero or negative

  ---
  Component Breakdown

  App.jsx                  — root; owns all state
  ├── IncomeInput.jsx      — controlled input for income
  ├── ExpenseForm.jsx      — form to add a single expense (name + amount)
  ├── ExpenseList.jsx      — renders the list of expenses
  │   └── ExpenseItem.jsx  — one row: name, amount, delete button
  └── Summary.jsx          — shows total expenses + colored balance

  No routing, no context, no external state library needed — everything lives in
   App and flows down as props.

  ---
  Data Structure

  // App state
  const [income, setIncome] = useState('');          // string → parsed to float
   on use

  const [expenses, setExpenses] = useState([]);       // array of expense 
  objects:
  // { id: crypto.randomUUID(), name: string, amount: number }

  Derived values (not stored in state — computed on each render):
  const totalExpenses = expenses.reduce((sum, e) => sum + e.amount, 0);
  const balance = parseFloat(income || 0) - totalExpenses;

  ---
  Build Order

  1. Scaffold state in App.jsx
  Add income and expenses state, plus addExpense and removeExpense handlers.
  Verify with console.log.

  2. Build IncomeInput
  Controlled <input type="number"> that updates income. Wire it to App.

  3. Build ExpenseForm
  Local state for name + amount inputs. On submit: call addExpense(name, 
  amount), clear inputs.

  4. Build ExpenseItem + ExpenseList
  ExpenseList maps over the array and renders an ExpenseItem per entry. Each
  item has a delete button that calls removeExpense(id).

  5. Build Summary
  Compute totalExpenses and balance. Apply a CSS class (positive / negative) to
  the balance element based on its sign.

  6. Style
  Wire up color classes in CSS (.positive { color: green }, .negative { color: 
  red }). Basic layout last so logic is solid first.

● Build is clean. Here's exactly where and how localStorage is used, all in
  App.jsx:

  ---
  How it works

  Reading on startup — lazy initializers (App.jsx:21–30)

  const [income, setIncome] = useState(
    () => localStorage.getItem(STORAGE_KEYS.income) ?? ''
  )
  const [expenses, setExpenses] = useState(() => {
    try {
      return JSON.parse(localStorage.getItem(STORAGE_KEYS.expenses) ?? '[]')
    } catch {
      return []
    }
  })

  The function passed to useState is a lazy initializer — React calls it exactly
   once, before the first render. This means the app reads from localStorage
  immediately at startup, so the UI never flashes empty and then fills in.
  Income is stored as a plain string so no parsing is needed. Expenses are
  stored as JSON, so JSON.parse converts the string back into an array of
  objects. The try/catch ensures a corrupt value (e.g. someone manually edited
  the storage) doesn't crash the app.

  Writing on change — effects (App.jsx:33–43)

  useEffect(() => {
    localStorage.setItem(STORAGE_KEYS.income, income)
  }, [income])

  useEffect(() => {
    localStorage.setItem(STORAGE_KEYS.expenses, JSON.stringify(expenses))
  }, [expenses])

  Each useEffect runs after React finishes rendering, but only when its
  dependency (income or expenses) has actually changed. Income is saved as-is.
  Expenses are serialized with JSON.stringify before saving since localStorage
  only stores strings.

  Storage keys (App.jsx:13–16)

  const STORAGE_KEYS = {
    income: 'smartbudget_income',
    expenses: 'smartbudget_expenses',
  }

  Both keys are defined once at the top of the file so there's no risk of a typo
   mismatch between the read and write calls.

● Clean. Here's the explanation:

  ---
  Why it was broken

  In JSX, writing `$${balance.toFixed(2)}` with a negative balance produces:

  balance = -150
  balance.toFixed(2)  →  "-150.00"
  `$${...}`           →  "$-150.00"  ✗

  The $ is inserted before the string, but the minus sign is already the first
  character of .toFixed(2)'s output — so the dollar sign ends up between the

  Bug 1 — Color index shift in BudgetPieChart when a slice is filtered out 
  (BalanceChart.jsx + BudgetPieChart.jsx)

  errors can build up. The result might be balance = -0.000000000000002 when the
   true value is $0.00. This pushes the balance into the red color class and the
   "negative" chart state even though it should display as exactly zero.
  .toFixed(2) in formatCurrency hides it in the display, but the >= 0
  comparisons see the raw value.
